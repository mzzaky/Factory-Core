Overview

Core System

The Factory System is the heart of the FactoryCore plugin. It allows players to own, manage, and operate physical factories directly inside the Minecraft world. Each factory is a persistent entity tied to a protected region, capable of consuming raw materials and producing finished goods through configurable recipes.

üí° What is a Factory?
A factory is a player-owned in-game structure bound to a WorldGuard region. It holds two separate inventories (input and output), runs production asynchronously in the background, and advances through a level system to unlock better efficiency and higher storage capacity.

The system integrates with the Economy (via Vault), the Employee System for workforce requirements, and the WorldGuard plugin for physical area protection ‚Äî forming a comprehensive industrial gameplay loop.

Architecture

Factory Model

Every factory in the system is represented as a rich data entity with the following core properties:

Property Description
ID & Name Unique system identifier and player-facing display name.
Region Bound to a WorldGuard region for physical area protection.
Factory Type Defines which product categories can be produced (e.g., Electronics, Food, Refinery).
Owner The UUID of the player with full administrative rights over the factory.
Level Progression tier that unlocks improved efficiency and larger storage.
Status Current operational state: STOPPED or RUNNING.
Status: Idle STOPPED Factory is ready but not producing. Awaiting a recipe start.
Status: Active RUNNING Production in progress. BossBar visible to the owner in real-time.
Processing Async Production runs in the background without blocking the server.

Storage System

Dual Storage

Each factory maintains two completely separate storage containers, managed by the internal StorageManager. This separation enforces a clear distinction between raw materials awaiting production and finished goods ready for collection.

üì•

Input Storage

Holds raw materials for production. Players must deposit items here before a recipe can start. The Deposit All feature makes bulk transfers quick and easy.

üì§

Output Storage

Receives finished goods automatically upon production completion. Players can withdraw results at any time. Capacity scales up with each factory level upgrade.

üì¶ Deposit All
The Deposit All shortcut allows players to transfer all compatible items from their inventory into the Input Storage in a single click, saving time when restocking factories with bulk materials.

Production Cycle

Operations

The production process follows a strict sequential flow across three phases. All state transitions are validated by the system before proceeding.

Phase 1 ‚Äî Initiation

  • 1
    Select a Recipe

    The player opens the factory GUI and chooses a recipe that matches the factory's type.

  • 2
    Employee Validation

    The system checks that at least one employee is assigned to this factory. Production is blocked if no employee is present.

  • 3
    Material Validation

    The system verifies that all required raw materials are present in the Input Storage in sufficient quantities.

‚ö†Ô∏è Employee Required
Production cannot start without at least one assigned employee. This is a hard system requirement. See the Employee System wiki page for details on hiring and assigning workers.

Phase 2 ‚Äî Processing

Once all validations pass, the factory enters the RUNNING state and processing begins:

  • Required materials are immediately consumed from Input Storage.
  • The factory status changes to RUNNING.
  • A BossBar progress indicator is displayed to the owner in real-time.
  • Production duration is calculated using the formula below, applying both level and employee bonuses.

Phase 3 ‚Äî Completion

When the production timer expires, the factory finalises the run:

  • Factory status reverts to STOPPED.
  • Finished products are automatically transferred to Output Storage.
  • Any console commands configured on the recipe are executed.
  • The owner receives a sound notification and a chat message confirming completion.

Upgrades

Progression

Players can invest in their factories by paying an upgrade cost to advance to the next level. Each level-up provides permanent, cumulative improvements.

üí∞

Progressive Cost

Upgrade fees scale with the current level using the formula: Base Price √ó 0.5 √ó Current Level, making early upgrades affordable and later ones a significant investment.

üì¶

Expanded Storage

Each upgrade increases the capacity of the Output Storage, letting factories accumulate more products between collection runs.

‚ö°

Faster Production

Higher levels grant a permanent production time reduction (Level Bonus), stacking multiplicatively with the Employee Buff for maximum throughput.

üìê Upgrade Cost Formula
Cost to upgrade = Base Price √ó 0.5 √ó Current Level. For example, upgrading a level 3 factory with a base price of $10,000 would cost $15,000.

Fast Travel

Each factory stores a designated Fast Travel Location. The owner can teleport instantly to their factory's location at any time ‚Äî a convenience feature especially useful for large maps with multiple factories spread across the world.

Data & Persistence

Storage Files

All factory data is persisted to disk in YAML format within the plugin's data directory. The system uses three separate files to logically partition different types of information.

plugins/FactoryCore/data/
1 factories.yml # Core factory data: location, level, owner, status
2 input-storage.yml # Raw material inventory per factory
3 output-storage.yml # Finished goods inventory per factory
File factories.yml Stores all factory metadata: world position, region name, type, level, owner UUID, and current status.
File input-storage.yml Tracks the contents of each factory's raw material inventory, keyed by factory ID.
File output-storage.yml Tracks the contents of each factory's finished goods inventory, keyed by factory ID.
üíæ Data Safety
Data is written to disk whenever a factory's state changes (production start, completion, upgrade, or storage modification). It is recommended to include these three YAML files in your regular server backup routine to prevent data loss.
